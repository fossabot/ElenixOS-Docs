# Script Native Interface (SNI)

Script Native Interface (**SNI**) is an interface for calling C functions in scripts.

## SNI Type Bridge Layer

### Basic Data Types

Basic data types include:
- Number
- Boolean
- String

SNI directly uses JerryScript's API to handle basic data types.

### Structs

SNI divides structs into two categories: Value Object and Handle Object.

#### Value Object

**Value Object** is a runtime object that does not independently own underlying resources, used to represent pure data or configuration values.

Value Object does not need to be explicitly created or destroyed, and its lifecycle is usually limited to a single function call or expression evaluation process. It is directly encapsulated as a JS object in the bridge layer, and its member variables are directly mapped to the properties of the JS object.
They are not registered or tracked by the runtime, nor do they participate in Realm-level resource management.

The lifecycle of Value Object is **Stack-Scoped**.

Stack lifecycle: Managed automatically by the JS engine, Value Object on the stack will be automatically destroyed when the function call ends.

#### Handle Object

**Handle Object** is a runtime object that represents a reference to underlying resources. It does not contain resource data itself, but serves as an indirect access entry to the underlying allocated entity.

Handle Object must be obtained through explicit creation and supports explicit destruction.
Its lifecycle is directly controlled by script logic, but the runtime provides unified fallback recycling when the Realm ends to prevent resource leaks.

The lifecycle of Handle Object is **Heap-Scoped**.

Heap lifecycle: Created explicitly by script logic, and underlying resources are released by calling specific destruction functions. When all references to the Handle Object are released, the runtime will automatically trigger its destruction process. In this system, objects with heap lifecycle will be uniformly destroyed when the Realm ends.

## SNI API Export Layer

The API Export Layer (**SNI API Export Layer**) is responsible for exporting APIs to the JS Realm, making them directly callable in scripts. APIs are defined through an API Description Table (**API Description Table**).

The description table includes:
- Functions
- Enums
- Constants
- Sub-namespaces

This layer is mainly responsible for API structure organization.

### API Description Table

The API Description Table is an array of C language structs, each element representing an API entry.

Each API entry contains the following fields:
- Name
- Type
- Value

:::danger
In the last element of the array, the name field must be NULL to identify the end of the array.
:::

**API Types**

Currently supported API types in SNI are:
- Functions: Function pointers of type `jerry_external_handler_t`
- Constants: Integer constants, floating-point constants, string constants
- Sub-entries: Pointers to sub-entries, used to implement recursive namespace structures

#### Function API

Function types are actually function pointers of type `jerry_external_handler_t`. You need to implement functions of this type to handle JS calls.

The parameters and return values of this function need to meet the requirements of the JerryScript engine. For details, you can refer to [jerry_external_handler_t - JerryScript Documentation](https://jerryscript.net/api-reference/#jerry_external_handler_t).

#### Constant API

Constant API refers to integer values, floating-point constants, and string constants defined in the description table, which are directly exported as constants in JS.
They are usually enum types or constants defined by macros in C code.

#### Sub-entry API

Sub-entry API refers to pointers to sub-entries defined in the description table, used to implement recursive namespace structures.

For example:

```c
jerry_value_t my_lv_obj_create(const jerry_call_info_t *call_info_p,
                               const jerry_value_t args_p[],
                               const jerry_length_t args_count)
{
    // Parameter checking
    // ...
    // Parameter type conversion
    // ...
    // Execute C function
    // ...
    // Check return value
    // ...
    // Return created Handle Object
}

const struct sni_api_entry_t lvgl_api_desc[] = {
    { "obj", SNI_ENTRY_NAMESPACE, { .sub_entries = lvgl_obj_api_desc } },
    // Other sub-entries...
};

const struct sni_api_entry_t lvgl_obj_api_desc[] = {
    { "create", SNI_ENTRY_FUNCTION, { .function = my_lv_obj_create } },
    // Other sub-entries...
};
```

In this way, you can call the C function `my_lv_obj_create()` through `lv.obj.create()` in JS.

### API Export Process

The API export process consists of the following steps:
1. Obtaining description table code
2. Parsing the description table
3. Mounting the API

#### Obtaining Description Table Code

Description table code is generally generated by Python scripts.

For example, the description table code for LVGL API can be generated with the following command:

```bash
python3 generate_lvgl_desc.py
```

Of course, you can also write description table code yourself according to your needs.

#### Parsing the Description Table

Parsing the description table only requires calling the `sni_api_build()` function to build the API description table. If the build is successful, it will return a JS object of type `jerry_value_t`, under which all API entries are mounted. This object is usually called the Global Native Capability Object, for example, `lv` is a global native object.

#### Mounting the API

The API mounting process is relatively simple, just call the `sni_api_mount()` function to mount the API description table to the specified JS Realm.

For example, mounting the LVGL API description table to a specified Realm can be implemented with the following code:

```c
jerry_value_t lvgl_api_obj = sni_api_build(lvgl_api_desc);
sni_api_mount(jerry_realm, lvgl_api_obj, "lv");
```

After successful mounting, the LVGL API can be directly called in the JS Realm.

For example:

```js
lv.obj.create();
```
