本指南将帮助你了解 ElenaOS 的开发流程、编码规范和贡献方式，为参与 ElenaOS 开发提供完整的指导。

```bash
git clone --recursive https://github.com/Sab1e-dev/ElenaOS.git
cd ElenaOS
```

## 编码规范

### C 语言编码规范

#### 命名约定

- **文件名**：使用小写字母和下划线，例如 `elena_os_app.c`
- **函数名**：使用小写字母和下划线，例如 `eos_app_install`
- **变量名**：使用小写字母和下划线，例如 `app_id`
- **常量名**：使用大写字母和下划线，例如 `EOS_APP_DIR`
- **类型名**：使用驼峰命名法，例如 `script_pkg_t`

#### 代码风格

- **缩进**：使用 4 个空格进行缩进
- **括号**：左括号与语句在同一行，右括号单独一行
- **行长度**：每行代码不超过 80 个字符
- **注释**：使用 Doxygen 风格的注释

#### 示例

```c
/**
 * @brief 安装应用
 * @param eapk_path eapk 安装包路径
 * @return eos_result_t 安装结果
 */
eos_result_t eos_app_install(const char *eapk_path) {
    // 函数实现
    return EOS_OK;
}
```

### JavaScript 编码规范

#### 命名约定

- **变量名**：使用驼峰命名法，例如 `appId`
- **函数名**：使用驼峰命名法，例如 `installApp`
- **常量名**：使用大写字母和下划线，例如 `MAX_APP_COUNT`
- **对象属性**：使用驼峰命名法，例如 `appName`

#### 代码风格

- **缩进**：使用 2 个空格进行缩进
- **括号**：左括号和右括号单独一行
- **行长度**：每行代码不超过 80 个字符
- **注释**：使用 JSDoc 风格的注释

#### 示例

```javascript
/**
 * 安装应用
 * @param {string} eapkPath - 应用包路径
 * @returns {number} 安装结果
 */
function installApp(eapkPath)
{
    // 函数实现
    return 0;
}
```

## 开发流程

### 分支管理

ElenaOS 使用 Git 进行版本控制，采用以下分支管理策略：

- **main**：主分支，包含稳定版本
- **develop**：开发分支，包含最新开发代码
- **feature/xxx**：功能分支，用于开发新功能
- **bugfix/xxx**： bug 修复分支，用于修复 bug

### 开发步骤

1. **创建分支**：从 develop 分支创建新的功能分支

```bash
git checkout develop
git pull
git checkout -b feature/your-feature
```

2. **编写代码**：实现功能或修复 bug

3. **测试代码**：确保代码能够正常编译和运行

4. **提交代码**：提交代码到本地仓库

```bash
git add .
git commit -m "feat: 实现新功能"
```

5. **推送分支**：将分支推送到远程仓库

```bash
git push origin feature/your-feature
```

6. **创建 Pull Request**：在 GitHub 上创建 Pull Request，描述功能或修复的内容

## 贡献流程

### 贡献方式

1. **报告问题**：在 GitHub Issues 中报告 bug 或提出功能请求
2. **提交代码**：通过 Pull Request 提交代码
3. **改进文档**：改进项目文档
4. **测试**：测试代码并提供反馈

### 代码审查

所有提交的代码都需要经过代码审查，确保代码质量和一致性。代码审查的重点包括：

- 代码风格是否符合规范
- 功能是否正确实现
- 代码是否安全
- 性能是否合理
- 文档是否完整

### 提交规范

提交消息应遵循以下格式：

```
type: subject

body (optional)
```

其中，type 可以是以下之一：

- **feat**：新功能
- **fix**：bug 修复
- **docs**：文档改进
- **style**：代码风格调整
- **refactor**：代码重构
- **test**：测试代码
- **chore**：构建或依赖更新

示例：

```
feat: 添加应用安装功能

实现了应用安装的核心逻辑，包括解析应用包、验证应用信息和部署应用文件。
```

### 单元测试

ElenaOS 使用 Unity 测试框架进行单元测试。测试文件位于 `tests` 目录。

### 集成测试

集成测试用于测试系统的整体功能，确保各个模块能够正常协作。

### 测试流程

1. **编写测试**：编写单元测试和集成测试
2. **运行测试**：运行测试确保代码功能正确
3. **覆盖率分析**：分析测试覆盖率，确保代码被充分测试

## 常见开发问题

### 编译错误

- **问题**：编译失败，提示缺少依赖
  **解决**：安装缺少的依赖，确保所有子模块都已正确克隆

- **问题**：交叉编译失败
  **解决**：检查交叉编译工具链是否正确配置，确保 CMake 能够找到工具链

### 运行时错误

- **问题**：运行时崩溃
  **解决**：使用调试工具分析崩溃原因，检查内存使用是否合理

- **问题**：应用无法运行
  **解决**：检查应用权限，确保应用符合 ElenaOS 的应用规范

### 性能问题

- **问题**：系统运行缓慢
  **解决**：优化代码，减少内存使用，避免不必要的计算

## 最佳实践

1. **模块化设计**：将功能划分为小的、可测试的模块
2. **代码复用**：避免重复代码，提取公共功能
3. **错误处理**：合理处理错误，确保系统稳定性
4. **文档先行**：在实现功能前先编写文档，确保功能设计清晰
5. **测试驱动**：编写测试用例，确保代码质量
