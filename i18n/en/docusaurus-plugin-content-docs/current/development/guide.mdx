## Coding Standards

### C Language Coding Standards

#### Naming Conventions

- **File names**: Use lowercase letters and underscores, e.g., `elena_os_app.c`
- **Function names**: Use lowercase letters and underscores, e.g., `eos_app_install`
- **Variable names**: Use lowercase letters and underscores, e.g., `app_id`
- **Constant names**: Use uppercase letters and underscores, e.g., `EOS_APP_DIR`
- **Type names**: Use camel case, e.g., `script_pkg_t`

#### Code Style

- **Indentation**: Use 4 spaces for indentation
- **Brackets**: Opening bracket on the same line as the statement, closing bracket on a separate line
- **Line length**: Each line of code should not exceed 80 characters
- **Comments**: Use Doxygen-style comments

#### Example

```c
/**
 * @brief Install application
 * @param eapk_path eapk installation package path
 * @return eos_result_t Installation result
 */
eos_result_t eos_app_install(const char *eapk_path) {
    // Function implementation
    return EOS_OK;
}
```

### JavaScript Coding Standards

#### Naming Conventions

- **Variable names**: Use camel case, e.g., `appId`
- **Function names**: Use camel case, e.g., `installApp`
- **Constant names**: Use uppercase letters and underscores, e.g., `MAX_APP_COUNT`
- **Object properties**: Use camel case, e.g., `appName`

#### Code Style

- **Indentation**: Use 2 spaces for indentation
- **Brackets**: Opening and closing brackets on separate lines
- **Line length**: Each line of code should not exceed 80 characters
- **Comments**: Use JSDoc-style comments

#### Example

```javascript
/**
 * Install application
 * @param {string} eapkPath - Application package path
 * @returns {number} Installation result
 */
function installApp(eapkPath)
{
    // Function implementation
    return 0;
}
```

## Development Process

### Branch Management

ElenaOS uses Git for version control, adopting the following branch management strategy:

- **main**: Main branch, contains stable versions
- **develop**: Development branch, contains latest development code
- **feature/xxx**: Feature branch, used for developing new features
- **bugfix/xxx**: Bug fix branch, used for fixing bugs

### Development Steps

1. **Create branch**: Create a new feature branch from the develop branch

```bash
git checkout develop
git pull
git checkout -b feature/your-feature
```

2. **Write code**: Implement features or fix bugs

3. **Test code**: Ensure the code can compile and run normally

4. **Commit code**: Commit code to local repository

```bash
git add .
git commit -m "feat: Implement new feature"
```

5. **Push branch**: Push the branch to the remote repository

```bash
git push origin feature/your-feature
```

6. **Create Pull Request**: Create a Pull Request on GitHub, describing the functionality or fix

## Contribution Process

### Contribution Methods

1. **Report issues**: Report bugs or feature requests in GitHub Issues
2. **Submit code**: Submit code through Pull Request
3. **Improve documentation**: Improve project documentation
4. **Test**: Test code and provide feedback

### Code Review

All submitted code needs to go through code review to ensure code quality and consistency. The focus of code review includes:

- Whether the code style conforms to the specifications
- Whether the functionality is correctly implemented
- Whether the code is secure
- Whether the performance is reasonable
- Whether the documentation is complete

### Commit Specifications

Commit messages should follow the following format:

```
type: subject

body (optional)
```

Where type can be one of the following:

- **feat**: New feature
- **fix**: Bug fix
- **docs**: Documentation improvement
- **style**: Code style adjustment
- **refactor**: Code refactoring
- **test**: Test code
- **chore**: Build or dependency update

Example:

```
feat: Add app install feature

Implemented the core logic for app installation, including parsing app packages, verifying app information, and deploying app files.
```

## Documentation Writing

### Documentation Structure

ElenaOS documentation is located in the `docs` directory and written in Markdown format. The documentation structure is as follows:

- **introduction.mdx**: Project introduction
- **architecture/arch.mdx**: System architecture
- **architecture/script_engine.mdx**: Script engine
- **getting_started/quick_start.mdx**: Quick start
- **getting_started/build.mdx**: Build and use
- **development/dev_tools.mdx**: Development tools
- **core/**: Core module documentation
- **js-api/**: JavaScript API documentation
- **development/**: Development-related documentation

### Documentation Standards

- Use Markdown format
- Use clear heading levels
- Code blocks use ``` markers and specify the language
- Charts use mermaid syntax
- Documentation should remain consistent with code

## Common Development Issues

### Compilation Errors

- **Issue**: Compilation fails, indicating missing dependencies
  **Solution**: Install missing dependencies, ensure all submodules are correctly cloned

- **Issue**: Cross-compilation fails
  **Solution**: Check if the cross-compilation toolchain is correctly configured, ensure CMake can find the toolchain

### Runtime Errors

- **Issue**: Runtime crash
  **Solution**: Use debugging tools to analyze the crash cause, check if memory usage is reasonable

- **Issue**: Application cannot run
  **Solution**: Check application permissions, ensure the application complies with ElenaOS application specifications

### Performance Issues

- **Issue**: System runs slowly
  **Solution**: Optimize code, reduce memory usage, avoid unnecessary calculations

## Best Practices

1. **Modular design**: Divide functionality into small, testable modules
2. **Code reuse**: Avoid duplicate code, extract common functionality
3. **Error handling**: Handle errors properly to ensure system stability
4. **Documentation first**: Write documentation before implementing functionality to ensure clear feature design
5. **Test-driven**: Write test cases to ensure code quality
